# 🎬 一键IP二创剧本+分镜表全流程创作技能

## 一、技能定位

这是一个从原始IP素材到最终飞书多维表格分镜表的**全流程自动化**技能。它把原本需要多次交互的复杂工作流压缩到一次完成，但每个环节依然需要用户确认关键决策。

理解了核心工作流的本质很重要。**这个技能的核心价值不是"自动化生成"，而是"将创作者的灵魂判断嵌入工业化流程"**——在需要人类审美决策的地方停下来让用户选择，在不需要的地方全速推进。

---

## 二、全流程概览

```
Step 1: 原始IP输入      ← 用户提供名称/梗概/已写剧本
       ↓
Step 2: 二创方向策划    ← 穷举5-8个方向供用户选择
       ↓
Step 3: 用户确认方向    ← 用户决定走哪条路
       ↓
Step 4: 写二创剧本      ← 按质量标准写剧本
       ↓
Step 5: 用户确认剧本    ← 用户提出修改意见或直接通过
       ↓
Step 6: 拆分为分镜表    ← 按15秒Batch规则拆分
       ↓
Step 7: 写入飞书多维表格 ← 自动化写入并提供链接
```

**核心理念**：每个步骤"写"的时候全速输出，"等"的时候只管等待。不要提前做下一步。

---

## 三、各步骤详细规范

### Step 1: 原始IP输入

用户可能提供以下三种形式的输入，区别处理：

**形式A：只说名称/梗概（如"太奶的股东会"）**
- 不要直接写，先问用户要具体内容
- 问："给我说说这讲的是一个什么故事？想二创成什么方向？"
- 或者如果用户给了简单的几句话梗概，基于这个展开

**形式B：提供了完整原始剧本**
- 直接进入Step 2

**形式C：提供了原始剧本+指定了二创方向**
- 跳过Step 2和3，直接进入Step 4

### Step 2: 二创方向策划 ⭐️ 最重要的环节

这是整个流程中最需要创造力的部分。用户信任你是顶级创作者才会让你策划。

**格式要求**：
穷举**5-8个**不同方向，每个方向必须：
1. **一句通俗易懂的解释**（一句话概括这个方向是什么意思）
2. **逻辑自洽的分析**（为什么这个方向适合这个原始IP）
3. **明确的二创定位**（情绪基调/目标受众/预期效果）

**推荐方向模板库**（根据原始IP类型灵活组合，不要生搬硬套）：

| 方向 | 适合场景 | 核心逻辑 |
|------|---------|---------|
| 类型片转译 | 正剧→悬疑惊悚/黑色喜剧/家庭伦理 | 保留核心冲突，换类型包装 |
| 时间线手术 | 线性叙事→倒叙/多线并进/循环 | 同一事件，不同编排顺序 |
| 叙事视角反转 | 主角视角→反派视角/旁观者视角 | 同一个故事，换个人讲完全不同 |
| 认知错位 | 信息不对称→制造戏剧性 | 观众知道但角色不知道的张力 |
| 象征道具接入 | 植入一个反复出现的符号 | 用具体物件串联全篇 |
| 时代/情境迁移 | 搬到截然不同的背景 | 把2025年的事搬到1990年 |
| 乱入/跨界 | 主角跑到另一个世界 | 反差混搭产生惊喜 |
| 双线并行 | 两条看似无关的线最后交汇 | 制造惊喜的关联 |
| 伏笔回收 | 挖坑埋线→后面全盘回收 | 早期细节在结局变成关键 |
| 情绪颠覆复调 | 笑中带泪/惊出冷汗/暖到发毛 | 混合相反情绪加深印象 |

**不要只列方向名称，每个方向都要给出有说服力的分析和落地可能**。用户不是专家，要靠你帮他们想象。每个方向写完之后，加一句具体的二创效果推测。

**示例输出结构**：

> **方向一：类型片转译（正剧→黑色职场喜剧）**
> 原始剧本偏正剧风，改写成黑色喜剧后，容小棉的"太奶气场"会变得更加荒诞搞笑。比如她进电梯那段可以加入更多反差笑点——大人们看到小孩进电梯的表情特写被拉长，配滑稽音效。适合25-35岁职场人群，解压+爽文并重。
>
> **方向二：时间线手术（倒叙开篇）**
> 直接从停车场密会的最高潮开场——黑色轿车接走容小棉，王建军的人和六岁小孩互作揖——然后黑屏字幕"24小时前"，倒叙回去。这一下子就把悬疑感拉到顶。适合喜欢悬疑反转的短剧受众。

每个方向的描述控制在60-100字左右，不要太长但也别太短。

**等用户选择后再进入下一步，不要替用户决策**。

### Step 3: 用户确认方向

- 用户选了一个或多个方向（如果选多个请用户确认以哪个为主）
- 如果有人想法方向融合，记录下来并确认
- 确认后进入Step 4

### Step 4: 写二创剧本

这是整个流程中最容易被低估的环节。剧本不是"故事概要"，而是**拿来就能直接分镜拍摄的详细脚本**。每个场景必须有足够的具体细节，让读者能在脑中直接看到完整的画面序列。

#### 前置准备：先问3个问题

不要直接写剧本。先向用户确认以下3个问题，等回答后再创作：

1. **风格**：选择（或自行描述）搞笑喜剧/文艺抒情/悬疑惊悚/热血武侠/科幻未来/温馨治愈/悲剧虐心/魔幻奇幻/黑色幽默/商业广告 等
2. **时长**：_____ 秒（如已指定则跳过）
3. **角色**：
   - 如果已有想法，用户提供：角色名+年龄+一句性格描述+一个外貌特征+一个口头禅/小动作
   - 如果没有，由你设计2-3个角色

#### 角色小传（每个角色必须包含）

每个角色写一段≥30字的人物小传，包含：
- **背景**：这个人从哪里来？什么身份？
- **性格**：一句话性格描述（不能只写"善良/勇敢"，要写"嘴硬心软，生气时会用扇子敲手心"这类具体特征）
- **核心欲望**：这个角色在整个故事里最想要什么？
- **口头禅/小动作**：标志性的语言或动作习惯（如"案子要紧"、"本宫看你是活腻了"）

示例：
> **临安公主**（22岁）
> 大奉王朝最受宠的公主，表面高贵优雅，实则内心戏超多。性格傲娇+毒舌+护短，核心欲望是想让许七安多陪陪她而不是总去破案。口头禅："本宫看你是活腻了"，小动作：生气时会用扇子敲手心。

#### 世界观/背景设定

一句话说明故事发生的独特环境。示例：
> 大奉王朝的京城，一个看似古代实则充满现代职场文化的奇妙世界。打更人相当于现代刑警，大理寺就是警察局。

尤其是"时代迁移"这类方向，世界观设定必须让读者迅速理解新背景的规则。

#### 质量标准

- **开头钩子**：前15秒必须有反常识画面——观众一上来就被抓住
- **三翻四抖**：同一剧情线至少3次情绪翻叠，最后1次轻抖收束
  - 第一翻：你不对劲（开局小异常）
  - 第二翻：你真不对劲（证据铺垫递进）
  - 第三翻：你完全不是人（核心反转）
  - 轻抖：原来早有伏笔（情感落点）
- **带钩结尾**：结尾必须打开新悬念，让观众想追下一集
- **节奏控制**：快(0-20%) → 中(20-50%) → 慢转极快(50-80%) → 缓(80-95%) → 极慢(95-100%)
- **无抽象大词**：禁止"坚持/梦想/温暖/美好/幸福/希望/永远"
- 每个场景20-30秒，全部场景加起来=用户指定时长±5秒
- 场景数量：根据总时长/25取整，通常8-12个场景
- **所有文本必须使用中文标点符号**
- **每个场景必须包含可拍摄的动作细节，不能是意识流**

#### 剧本正文格式（每个场景必须包含）

1. **场景编号与具体地点**：如"公主府·清晨""大理寺·上午"
2. **氛围词**：温度、光线、气味、声音，至少2个感官维度
3. **画面动作**：不得出现"开心/难过/炫耀"等情绪词直接修饰动作。必须用**可拍摄的动作序列**描述。例如：不能说"他很紧张"，要说"他反复拧着衣角，额头渗出汗珠"。**具体的动作序列比任何形容词都更有价值**。
4. **对白**：必须符合角色性格，且**至少有1句潜台词**——话里有话，不能直说"我爱你""我恨你""我要成功"
   - **潜台词示例**：不说"我很担心"，说"你要是死了，本宫上哪儿去找这么好用的刀子？"
   - **重复台词变调**：可用同一句台词在不同场景中含义逆转，如开头"案子要紧"是抱怨，结尾"案子要紧"是心疼
5. **建议时长**：精确到秒，所有场景时长总和=用户指定时长±5秒

#### 额外硬性要求

- **60%进度转折点**：剧本必须在60%进度处有一个明确转折（反转/意外/情绪陡变）
- **意外波动细节**：必须包含一个让观众情绪意外波动的设计，可以是：
  - 意想不到的道具（如生日宴会上拿出一个空盒子）
  - 重复出现但含义变化的台词（如"没事"说了三次，最后一次哭了）
  - 对称但结果不同的动作（如开头摔门而出，结尾轻轻关门）
- **结尾余味**：最后一个画面必须是一个**有意味的特写**（眼神/物品/环境），让观众看完还想
- **配乐/音效建议**：每个场景标注背景音乐的情绪关键词（如"紧张电子乐""温柔钢琴""无声张力"）

#### 写完后的自检

- 是否有完整的角色小传？每个角色≥30字？
- 是否有世界观设定？
- 每个场景是否有氛围词？≥2个感官？
- 画面动作是否可拍摄？是否禁用了情绪修饰词？
- 对白是否有潜台词？
- 是否有60%转折点？
- 是否有意外波动细节？
- 是否含配乐建议？
- 结尾是否有余味特写？
- 是否禁止了抽象大词？

**写完剧本后，发给用户确认。等用户确认或提修改意见后再进Step 5。不要直接推进分镜表。**

### Step 5: 用户确认剧本

- 用户可能提出修改意见、增减场景、调整人设
- 按意见修改后再次确认
- 确认后进入Step 6

### Step 6: 拆分为分镜表

这是最技术性的步骤，需要严格按照规则执行。

**分镜拆解规则（硬性）**：

#### 1. 拆解流程

```
输入：已确认的二创剧本（N个场景）
输出：结构化分镜数据（N个场景×M个镜头）

流程：
For each 场景:
  1. 按场景内自然事件流确定镜头数量
  2. 分配镜头时长（每个镜头2-6秒）
  3. 检查是否超过15秒Batch限制
  4. 如果超过15秒，将满15秒后的镜头划分到下一个Batch
  5. 确定每个Batch内的分镜号和镜头号
  6. 确定每个镜头的时间段
  7. 填写所有21列字段
```

#### 2. 镜号规则（硬性）

镜号不能为小数，必须是整数。格式：「分镜号-镜头号」，如1-1,1-2,2-1,2-2

- 分镜号 = 该场景内的第几个15秒Batch（从1开始）
- 镜头号 = 该Batch内的第几个镜头（从1开始）

示例：场景1有8个镜头
- Batch1: 镜头1(2秒) 1-1, 镜头2(3秒) 1-2, 镜头3(3秒) 1-3, 镜头4(4秒) 1-4, 镜头5(3秒) 1-5（累积13秒，不满15）
- Batch2: 镜头6(3秒) 2-1, 镜头7(4秒) 2-2, 镜头8(3秒) 2-3（累积10秒）

#### 3. 15秒Batch规则（硬性）

- 每个Batch内累计时长≤15秒
- 满15秒后，下一个Batch重新从0开始计时
- Batch内最后一个镜头结束时如果恰好满15秒，下一个Batch从0开始
- 分镜时间段不能出现≥15的数字
- 分镜时间段的计算方式：前一个数字=前一个镜头的结束时间，后一个数字=累积到该Batch的总时长

注意：Batch的编号在每个场景内是独立的。场景1的Batch1和场景1的Batch2组成了场景1的所有镜头。场景2重新从Batch1开始。

#### 4. 21列字段定义

| 序号 | 字段名 | 类型 | 填写规范 |
|------|--------|------|---------|
| 1 | 集数 | 文本 | 默认"1" |
| 2 | 场次 | 文本 | 按剧本场景号，1,2,3... |
| 3 | 镜号 | 文本 | 格式「分镜号-镜头号」，如1-1 |
| 4 | 镜头时长 | 文本 | 格式"2秒""3秒"等 |
| 5 | 分镜时间段(s) | 文本 | 格式"0-2""2-5"等 |
| 6 | 镜头角度 | 文本 | 平视/俯视/仰视/低角度/鸟瞰 |
| 7 | 景别 | 文本 | 全景/中景/近景/特写/大特写 |
| 8 | 运镜 | 文本 | **必须差异化设计！运镜方式≥6种，同一运镜不连续出现2次。详见下方"运镜设计规则"** |
| 9 | 特殊速度 | 文本 | 默认"-"，有需求填"慢动作0.5x"等 |
| 10 | 转场方式 | 文本 | 切/淡入淡出/跳切/溶解/叠化/黑场 |
| 11 | 画面内容描述 | 文本 | **重中之重：详细、可拍摄的动作描述，含人物位置、动作顺序、环境细节、小道具，50字+，使用中文标点** |
| 12 | 场景中的人物 | 文本 | 出现在这个镜头里的所有人物 |
| 13 | 角色表情/情绪 | 文本 | 每个角色的情绪状态，格式"角色A情绪/角色B情绪" |
| 14 | 音色备注 | 文本 | 默认"默认" |
| 15 | 光源/色调 | 文本 | 环境光+色调，如"自然晨光/暖色调""会议室冷白灯光" |
| 16 | 特效/合成说明 | 文本 | 无特效填"-" |
| 17 | 关键帧图片提示词 | 文本 | **重中之重：40字+，格式「主体动作/环境/光线/材质/风格/画质/8k」，必须包含写实风格、电影级、8k等画质词** |
| 18 | 台词 | 文本 | 对白内容，无台词填"-" |
| 19 | 音效/BGM | 文本 | 镜头需要的音效 |
| 20 | 导演意图 | 文本 | 一句话说明叙事目的 |
| 21 | 状态 | 文本 | 默认"待生成" |

**画面内容描述写作要点**：
"可拍摄"是唯一的判断标准。不要写"她很难过"，要写"她低头看桌面，睫毛垂下来，手指在桌沿无意识地来回摩擦"。**具体的动作序列**比形容词更有价值。禁用"开心地/难过地/炫耀地"等情绪词直接修饰动作——让动作本身传递情绪，而不是打标签。

**关键帧图片提示词写作要点**：
提示词的核心是**让AI视频模型能生成准确的画面**。格式上，用斜杠分隔不同要素比逗号更清晰：
`主体动作/环境/光线氛围/材质细节/画风/画质词`
不需要完整英语句子，关键词堆叠就够。但必须包含足够的视觉信息，不能只有"小女孩在会议室"这种级别。

**运镜设计规则（非常重要）**

运镜不能随机填"固定"，必须根据每个镜头的画面内容适配。理解运镜的核心逻辑比记住规则列表更重要：**运镜是镜头语言的一部分，它引导观众看什么、怎么看、感受到什么情绪。** 同一个画面内容，换一种运镜就完全改变了情感基调。

以下是不同运镜的适用场景和设计理由：

| 运镜方式 | 适用场景 | 设计理由 |
|---------|---------|--------|
| **慢推** | 情绪递进、对话关键点、表情变化 | 慢慢推向人物或物件，让观众注意力聚焦，暗示"请注意这里很重要" |
| **跟拍** | 人物移动、跑动、走位 | 让观众代入移动者视角，增强沉浸感，展示动作的连续性 |
| **平移** | 环境交代、全景展现、空间关系 | 从左到右或从上到下展示环境全貌，让观众理解空间布局 |
| **慢拉** | 从近景展开、展示空间关系 | 从一个细节慢慢拉开，揭示更多信息，制造"原来是这样"的感觉 |
| **手持晃动** | 慌乱、紧张、突发事件 | 模拟人手持相机的不稳定感，传递慌乱不安的情绪 |
| **快速摇移** | 突然出现的人物、动作打断 | 快速从一个对象移到另一个对象，制造"出乎意料"的冲击感 |
| **慢移** | 跟随动作、温柔时刻 | 同跟拍但速度更慢，适合温馨、细腻的动作 |
| **固定** | 喜剧笑点、情绪特写、对话戏 | 让画面静止，观众专注看表情或笑点，不被打断 |
| **俯视移** | 全景+空间展示 | 俯拍+移动，同时展示人物和环境，常用于情景喜剧的开场 |
| **低角度固定** | 角色权威感/弱势感 | 从下往上拍，让角色显得高大或渺小 |

**设计原则**：
- 同一个场景内的运镜要交替变化，不能连续2个镜头用同一运镜
- 全景/环境交代镜头用平移/慢拉，情绪镜头用慢推/固定
- 喜剧笑点用固定（让观众看清），惊吓/紧张用手持晃动或快速摇移
- 开场镜头通常用慢推或淡入，高潮用快速或晃动，结尾用固定或叠化
- 总共使用的运镜类型不少于6种，避免单调

### Step 7: 写入飞书多维表格

**⚠️ 切记：必须使用用户身份创建多维表格，不能用机器人身份。**

#### 7.1 创建多维表格

```bash
lark-cli base +base-create --name '<剧名>——分镜表'
```

```json
  {"field_name":"场次","type":"text"},
  {"field_name":"镜号","type":"text"},
  {"field_name":"镜头时长","type":"text"},

  {"field_name":"镜头角度","type":"text"},
  {"field_name":"景别","type":"text"},
  {"field_name":"运镜","type":"text"},
  {"field_name":"特殊速度","type":"text"},
  {"field_name":"转场方式","type":"text"},
  {"field_name":"画面内容描述","type":"text"},
  {"field_name":"场景中的人物","type":"text"},
  {"field_name":"角色表情/情绪","type":"text"},
  {"field_name":"音色备注","type":"text"},
  {"field_name":"光源/色调","type":"text"},

  {"field_name":"状态","type":"text"}

```

保存为 `fields_21.json` 在当前目录。

#### 7.3 创建带字段的表

```bash
lark-cli base +table-create --base-token <token> --name '分镜表' --fields @fields_21.json
```

如果 `--fields` 报错，检查字段JSON中type是否用了字符串（"text"）而不是数字（1）。

#### 7.4 准备批量写入数据

生成批量写入JSON文件。格式为：
```json
{
  "fields": ["集数","场次","镜号","镜头时长","分镜时间段(s)",...],
  "rows": [
    ["1","1","1-1","2秒","0-2","平视","全景","跟拍","-","淡入","画面描述...","容小棉","专注","默认","自然晨光/暖色调","-","提示词...","-","音效...","导演意图...","待生成"],
    ...
  ]
}
```

建议用Python脚本生成（先写入Python文件再执行），避免cmd中编码问题。

#### 7.5 批量写入

```bash
lark-cli base +record-batch-create --base-token <token> --table-id <tableId> --json @batch_data.json
```

写入成功后检查 `record_id_list` 长度是否匹配数据条数。

#### 7.6 数据修正策略

如果写入后发现问题（如运镜错误、描述不准确等）：

**推荐做法**：删表重建，一次性写入正确数据（最快）
```bash
lark-cli base +table-delete --base-token <token> --table-id <tableId> --yes
lark-cli base +table-create --base-token <token> --name 分镜表 --fields @fields_21.json
lark-cli base +record-batch-create --base-token <token> --table-id <newTableId> --json @correct_data.json
```

**不推荐的做法**（都比较慢或有坑）：
- ❌ `+record-upsert` 逐条更新 — 23条需要23次API调用
- ❌ `+record-batch-update` — 只能对所有记录设同一个值，不能逐条不同
- ❌ `feishu_bitable_update_record` — 需要用户OAuth授权，经常403
- ❌ `feishu_bitable_app_table_record` batch_update — 也需要用户OAuth授权

#### 7.7 提供链接

最终提供：
```
https://<域名>.feishu.cn/base/<app_token>
```

## 四、Windows兼容注意事项

本技能可能在Windows环境执行，需要特别注意：

1. **Python路径**：用 `python` 或 `py`，不用 `python3`
2. **CMD转义**：不用单引号，双引号内嵌双引号用 `""` 转义
3. **文件路径**：用反斜杠 `\` 或绝对路径
4. **命令串联**：PowerShell中不用 `&&` 和 `||`，分行执行或用分号
5. **JSON文件优先**：所有带中文的参数尽量用 `--json @file.json` 方式传递
6. **编码问题**：PowerShell直接执行带中文的`.py`文件 或 `-c` 参数时会炸（文件被转为GBK/引号被吃掉）。
   - ✅ **正确路径**：用PowerShell `.NET` API写UTF-8无BOM JSON文件 → `cmd /c "lark-cli ... @file.json"` 写入
   - ✅ **备用路径**：如果只需要写少量记录，用 `feishu_bitable_app_table_record.batch_create` 工具（需用户OAuth授权）
   - ❌ **不要尝试**：`write`工具写中文.py → exec执行、`python -c` 传中文参数、PowerShell内嵌双引号+"&&"串联
7. **写入前确认权限**：如果 `feishu_bitable_app_table_record.batch_create` 报 `need_user_authorization`，直接走lark-cli路径，不要绕弯子试其他API

## 六、常见问题

**Q: 为什么不用 `feishu_bitable_create_app` 建表？**
A: 该API以机器人身份创建多维表格，用户没有管理权限，后续无法写入数据也删除表。必须用 `lark-cli base +base-create` 以用户身份创建。

**Q: 为什么字段 type 要用字符串不是数字？**
A: `lark-cli base +table-create --fields` 接受的字段定义中，type字段使用字符串枚举值（"text"、"number"、"select"等），不是数字。

**Q: 写入后发现数据有误怎么办？**
A: 最快的方式是删表重建（见7.6数据修正策略）。不要试图逐条更新——那比删表重建慢10倍，而且容易遇到权限问题。

**Q: 写入成功了但数据不完整？**
A: 检查record_id_list的长度是否匹配预期条数。如果少了，可能是batch JSON格式问题，检查fields和rows的对应关系。

**Q: 为什么我的剧本被用户说"太简略"？**
A: 剧本不是"故事概要"，而是"直接可拍摄的详细脚本"。确保每个场景都有：具体的动作序列、潜台词对白、配乐建议；确保每个角色都有完整的小传。见Step 4的完整格式规范。

**Q: 为什么用户说"运镜全是固定"？**
A: 因为偷懒了。每个镜头的运镜都要根据画面内容独立设计——全景用平移/慢拉，情绪用慢推/固定，笑点用固定，紧张用手持/摇移。详见"运镜设计规则"。

## 七、注意

这个技能是一个全流程工具，**每次使用都有明确的分步输出**。不要让用户等太久——在需要用户确认的环节（Step 2、Step 3、Step 5）主动询问，其他环节全速执行。写完一个步骤后，用简洁的一句话告知当前状态，然后等待用户的下一步指令。

**重要经验总结**：
1. **剧本≠故事梗概**。剧本是分镜的输入材料，细节越丰富、分镜越好写。偷懒的后果是用户退货重写，反而更慢。
2. **运镜必须差异化设计**。所有镜头"固定"=不合格。根据每个镜头的画面内容适配不同运镜，类型≥6种，不连续重复。
3. **写入数据出错时删表重建比逐条修改快10倍**。不要试图通过API逐条更新23条以上的数据。
4. **Windows环境记得用 `cmd.exe /c "..."` 而非原生PowerShell**，否则 `@`、`&&`、中文都会出问题。
5. **编码是N地雷——走对路径一次过**。Windows下写中文文件绕过硬编码：用PowerShell的 `.NET UTF8无BOM API` 写JSON文件 → `lark-cli @file.json` 写入。绝对不要试图用 `cmd /c "python -c ..."` 或 `write` 工具写中文.py再exec，都会因GBK编码问题炸掉。JSON文件做中间载体是最稳的路线。
